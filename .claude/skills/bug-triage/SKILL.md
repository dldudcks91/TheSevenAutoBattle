## Phase 1: 버그 목록 로드

`production/qa/bugs/` 또는 `production/qa/bugs.md`에서 버그 파일을 읽는다.
버그 파일이 없으면 없다고 보고하고 종료한다.

---

## Phase 2: 버그 분류

각 버그에 대해:

### 심각도 (영향도)
| 등급 | 정의 |
|------|------|
| **S1 — 치명** | 게임 크래시, 데이터 손실, 전체 기능 불가 |
| **S2 — 높음** | 주요 기능 고장이지만 플레이 가능 |
| **S3 — 중간** | 기능 저하, 우회 방법 존재 |
| **S4 — 낮음** | 시각적 결함, 텍스트 오류, 게임플레이 영향 없음 |

### 우선순위 (긴급도)
| 등급 | 정의 |
|------|------|
| **P1 — 즉시 수정** | QA 블로킹, 출시 블로킹, 지난 스프린트 회귀 |
| **P2 — 빠른 수정** | 다음 마일스톤 전에 해결 필요 |
| **P3 — 백로그** | 고치면 좋지만 블로킹 아님 |
| **P4 — 보류** | 수용 리스크 또는 현재 범위 외 |

---

## Phase 3: 트렌드 분석

- 같은 시스템에서 3개 이상 버그 → 설계/구현 품질 문제 경고
- 완료된 기능에서 버그 → 회귀 경고

---

## Phase 4: 트리아지 보고서 출력

```
## 버그 트리아지 보고서

### 요약
| 우선순위 | 건수 |
|---------|------|
| P1 — 즉시 수정 | [N] |
| P2 — 빠른 수정 | [N] |
| P3 — 백로그 | [N] |
| P4 — 보류 | [N] |

### P1 버그 — 즉시 수정
| ID | 시스템 | 심각도 | 내용 |
|----|--------|--------|------|

### 시스템 이상 패턴
[패턴 발견 시 기술, 없으면 "없음"]

### 권장 액션
1. [가장 긴급한 행동]
2. [두 번째 행동]
```

## Phase 5: 저장

"이 트리아지 보고서를 `production/qa/bug-triage-[날짜].md`에 저장할까요?" 확인 후 저장.
