## Phase 1: 모드 결정

- `new` → 빈 플레이테스트 보고서 템플릿 생성
- `analyze [경로]` → 날 것의 메모를 읽어 구조화된 보고서로 변환

---

## Phase 2A: 새 템플릿 모드

아래 템플릿을 출력한다:

```markdown
# 플레이테스트 보고서

## 세션 정보
- **날짜**: [날짜]
- **빌드**: [버전/커밋]
- **플레이 시간**: [시간]
- **테스터**: [이름/ID]
- **테스트 유형**: [첫 플레이 / 복귀 / 타겟 테스트]

## 테스트 포커스
[어떤 기능이나 흐름을 테스트했는지]

## 첫인상 (처음 5분)
- **목표 이해?** [예/아니오/부분적]
- **조작 이해?** [예/아니오/부분적]
- **감정 반응**: [몰입/혼란/지루함/불만/흥미]

## 게임플레이 흐름
### 잘 된 부분
- [관찰 1]

### 불편 지점
- [문제 1 — 심각도: 높음/중간/낮음]

### 혼란 지점
- [플레이어가 혼란스러워한 부분과 이유]

### 즐거움 포인트
- [플레이어가 즐거워하거나 놀란 순간]

## 버그 발견
| # | 설명 | 심각도 | 재현 가능 |
|---|------|--------|----------|

## 기능별 피드백
### [기능명]
- **목적 이해?** [예/아니오]
- **재미있었나?** [예/아니오]
- **제안**: [테스터 의견]

## 전체 평가
- **다시 플레이하고 싶나?** [예/아니오/아마도]
- **난이도**: [너무 쉬움 / 적당 / 너무 어려움]
- **페이싱**: [너무 느림 / 적당 / 너무 빠름]

## 이번 세션 Top 3 우선순위
1. [가장 중요한 발견]
2. [두 번째 우선순위]
3. [세 번째 우선순위]
```

---

## Phase 2B: 분석 모드

날 것의 메모를 읽고 기존 기획 문서와 대조한다.
템플릿에 구조화된 내용을 채워 넣는다.
기획 의도와 충돌하는 플레이테스트 관찰을 플래그한다.

---

## Phase 3: 액션 라우팅

모든 발견을 4가지 버킷으로 분류:

- **기획 변경 필요** → `/design-review` 실행 권장
- **밸런스 조정** → `/balance-check` 실행 권장
- **버그** → `production/qa/bugs/`에 기록 권장
- **폴리시** → 백로그에 추가

---

## Phase 4: 보고서 저장

"이 플레이테스트 보고서를 `production/qa/playtests/playtest-[날짜].md`에 저장할까요?" 확인 후 저장.
