## Phase 1: 질문 정의

인자에서 컨셉을 파악한다. 이 프로토타입이 답해야 할 **핵심 질문**을 명확히 한다.
컨셉이 모호하면 질문을 먼저 명확히 한다 — 질문 없는 프로토타입은 시간 낭비다.

예시:
- "오토배틀 전투 속도가 재미있게 느껴지는가?"
- "죄종 시너지 3개가 쌓였을 때 효과가 체감되는가?"
- "코스트 한도 시스템이 직관적으로 이해되는가?"

---

## Phase 2: 프로젝트 컨텍스트 로드

`CLAUDE.md`와 `docs/GAME_DESIGN.md`를 읽어 현재 기술 스택과 게임 방향을 파악한다.

---

## Phase 3: 프로토타입 계획

3~5개 항목으로 최소 가능한 프로토타입 정의:

- 핵심 질문은 무엇인가?
- 답을 얻기 위한 최소한의 코드는?
- 생략 가능한 것은? (에러 처리, 폴리시, 아키텍처)

계획을 먼저 보여주고 확인을 받는다.

---

## Phase 4: 구현

"prototypes/[컨셉명]/ 에 프로토타입을 만들어도 될까요?" 확인 후 진행.

모든 파일 상단에 반드시 포함:
```
# PROTOTYPE - NOT FOR PRODUCTION
# Question: [검증할 핵심 질문]
# Date: [날짜]
```

완화된 기준 적용:
- 수치 하드코딩 허용
- 플레이스홀더 에셋 사용
- 에러 처리 생략
- 가장 단순한 접근법 사용
- production 코드 복사 허용 (import 금지)

---

## Phase 5: 프로토타입 보고서

```markdown
## 프로토타입 보고서: [컨셉명]

### 가설
[기대했던 것]

### 접근
[만든 것, 걸린 시간, 취한 단축]

### 결과
[실제 일어난 것 — 구체적으로]

### 측정 데이터
- 반응 속도: [해당 시]
- 체감 평가: [구체적으로 — "200ms 딜레이가 느리게 느껴짐" 수준]
- 반복 횟수: [몇 번 만에 동작]

### 권고: [PROCEED / PIVOT / KILL]

[근거와 함께 한 문단]

### 진행한다면
[프로덕션 구현을 위해 바꿔야 할 것]

### 교훈
[다른 시스템이나 향후 작업에 영향을 주는 발견]
```

---

## 중요 제약

- 프로토타입 코드는 절대 production 소스 파일에서 import 금지
- Production 코드는 절대 prototypes/ 에서 import 금지
- PROCEED 권고 시, 프로덕션 구현은 처음부터 새로 작성
- 총 작업량은 1~3일 분량으로 타임박스
