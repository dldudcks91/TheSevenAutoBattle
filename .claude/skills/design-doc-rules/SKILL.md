## 핵심 규칙: 기획서에는 데이터(숫자) 적지 않기

기획 문서에는 **의도·방향·시너지 컨셉·미결 항목**만 적는다.
**구체 수치(HP, 공격력, 사거리, 가격, 증가량, 스택 캡, 코스트 등)는 절대 넣지 않는다.**
수치는 데이터 파일(`src/data/units/*.tres`, `src/economy/run_state.gd` 의 const, 향후 CSV 등)에서만 관리한다.

### Why
같은 값이 기획 문서와 데이터 파일 양쪽에 적히면, 둘이 어긋났을 때 어느 게 진실인지 매번 헷갈린다. 밸런스 패치 한 번에 두 곳을 동시에 고쳐야 하고, 시간이 지나면 동기화가 무너져 잘못된 정보가 남는다 (사용자가 직접 겪고 학습한 페인포인트).

### How to apply

**기획 문서를 작성·수정할 때:**
- 숫자 리터럴이 들어가려 하면 멈추고 "이건 데이터 파일에 있어야 한다"고 판단
- 예시·구조도에서도 `체력 100 / 공격 +15` 같은 표현 금지 → "체력·공격 상향" 같은 방향성 표현으로
- 시너지·업그레이드 설명도 `X 스택마다 +Y%` 가 아니라 "킬 누적으로 공격력 폭증" 같이 의도만
- 사용자가 기획서에 숫자를 넣자고 하면 한 번 확인하고, 의도가 명확하면 데이터 파일 쪽에도 같이 반영하도록 제안

**기획 문서를 리뷰할 때:**
- 본문에 숫자 리터럴이 있는지 스캔 → 발견되면 의도 표현으로 바꾸도록 제안
- "수치/공식 정의 여부"를 완성도 체크 항목으로 쓰지 말 것 (data와 design doc의 책임이 섞이는 결과를 낳음)

### 예시

❌ 금지:
```
[Soldier] 기본 (체력 100 / 공격 10 / 코스트 2)
   ├─ [A] 돌격형 — 체력 +20, 공격 +15
   └─ [B] 방어형 — 체력 +40, 방어막 추가
```

✅ 허용:
```
[Soldier] 기본
   ├─ [A] 돌격형 — 체력·공격 상향, 위험 부담 감수
   └─ [B] 방어형 — 체력·생존성 상향, 받쳐주는 역할
```
